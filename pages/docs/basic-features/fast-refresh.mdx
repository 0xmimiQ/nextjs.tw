「快速重整」是 Next.js 的讓編輯的 React 組件獲得立即回饋的一個功能。**9.4 或更新的版本**中，這個功能在所有的 Next.js 應用程式是默認開啟的。隨著這個功能的開啟，幾乎所有的編輯都不會遺失組件的狀態而被快速地看見。

## 怎麼做到的？

- 如果你編輯的檔案**只有匯出 React 組件**，「快速重整」功能將只會更新該檔案，然後重新渲染你的組件。你可以在該文件中編輯，包括樣式、渲染邏輯、事件處理或特效。
- 如果你編輯的檔案內要匯出的 _不是_ React 組件，快速重整將會重新執行兩個檔案，並且引入它其他的。
- 最後，如果你**編輯的檔案**是引入自**非屬於 React 樹**的檔案，「快速重整」**將會執行完全載入**。你可能會有檔案是負責渲染一個 React 組件，但同時也匯出從**非屬於 React 組件**引入的值。例如：也許你的組件會匯出一個常數，和一個非 React 的公用程式檔案則引入它；在這樣的狀況下，可以考慮將該常數放置在另外一個檔案，然後引入常數到這兩個檔案。這將會重新啟用「快速重整」，其他的狀況通常也可以透過這樣的方式解決。

## 錯誤恢復

### 語法錯誤

如果你在開發時不小心有一個語法錯誤，你可以修正並再一次儲存。此錯誤將會自動地消失，你將不需要重新載入應用程式，同時也**不會遺失組件狀態**。


### 運行錯誤

如果犯的錯誤導致在組件內產生一個運行錯誤，你將會看到一個上下文覆蓋提醒。修復錯誤後將會自動地忽略該提醒而不會重新載入應用程式。
如果重新渲染期間沒有發生錯誤，組件狀態將會被保留。如果發生錯誤，React 會將更新後的程式碼重新安置在應用程式上。


如果你的應用程式內有[錯誤邊界](https://reactjs.org/docs/error-boundaries.html)，對於生產階段來說若能有容錯空間是一個不錯的主意，它們會在渲染錯誤後，將針對下一次的修正再次嘗試渲染。這意味著具備「錯誤邊界」可以預防經常性地重置到根部應用程式狀態。然而，要記住的是，「錯誤邊界」不該太過著重在細節。它們在生產階段被 React 使用，且必須是刻意被設計的。

## 限制

「快速重整」嘗試保留你所編輯的本地 React 狀態，但這僅在一切是安全的前提之下才能進行。以下是你每次編輯檔案時都會重置本地狀態的可能原因：

- class 組件不會保留本地狀態 (只有函式組件和 Hooks 才會)。
- 你所編輯的檔案除了 React 組件外，也許有 _其他_ 匯出。 
- 有時候，一個檔案會匯出呼叫高階組件的結果，如 `HOC(包裝組件)`；如果返回的組件是一個 class，它的狀態將會被重置。
- 匿名箭頭函式如 `export default () => <div />;` 將導致「快速重整」以致於不會保留本地的組件狀態。針對大型的程式碼庫，你可以使用我們的 [`name-default-component` codemod](/docs/advanced-features/codemods.md#name-default-component)。

隨著你的程式庫越來越多轉移至使用 函式組件和 Hooks，你可以期待在更多情境下狀態是被保留的。

## 提示

- 「快速重整」在函式組件(和 Hooks)下，默認是會保留 React 本地狀態。
- 有時候你可能會希望 _強制_ 重置狀態，然後使組件重新被掛載。例如：如果你想調整一個只會在掛載時才會顯現的動畫，你可以在編輯的檔案的任一位置加入 `// @refresh reset`，這對於檔案來說是本地指令，它在每次編輯時指示「快速重整」重新掛載檔案內被定義的組件。
- 你可以在開發過程中，放置 `console.log` 或 `debugger` 在你所編輯的組件內。

## 「快速重整」和 Hooks

可行的狀況下，「快速重整」會企圖在編輯之間保留組件的狀態。特別是 `useState` 和 `useRef` ，只要不去改變它們的參數或者是 Hook 呼叫的順序，它們保留的是之前的值。

包含相依性的 Hooks，如 `useEffect`、`useMemo` 和 `useCallback` 在「快速重整」期間會 _一直_ 更新。它們的相依性在重整期間將會被忽略。

舉例來說，當你更改這個 `useMemo(() => x * 2, [x])` 到 `useMemo(() => x * 10, [x])`，這將會導致重新運行，即使 `x`(相依性)並沒有改變，如果 React 不這樣做，你的編輯將無法呈現在螢幕上。

有時候，這將導致不可預料的結果，例如：即使是帶有一個空陣列作為相依性的 `useEffect` 也會在「快速重整」期間重新運行一次。
